## 变更内容

### 1. 支出审核模块

**背景**: 支出记录缺少审核流程，无法控制支出的合规性。

**改动**：
- FinanceExpense 模型新增 `approval_status`（审核状态：待审核/已通过/已驳回）、`approver`（审核人）、`approve_time`（审核时间）字段
- 新建 `backend/routers/finance_review.py` — 4 个 API 端点：统计卡片、筛选表格、通过操作、驳回操作
- 新建 `frontend/templates/finance_review.html` — 状态筛选 Tab + 搜索 + 驳回弹窗
- 新建 `frontend/static/js/finance_review.js` — 审核操作逻辑
- Alembic migration：`add_approval_fields_and_budget`

### 2. 预算管理模块

**背景**: 缺少预算编制和实际执行对比功能，经营决策缺乏数据支撑。

**改动**：
- 新增 `Budget` 模型（预算编号/月份/类别/类型/金额/备注）
- 新建 `backend/routers/finance_budget.py` — 6 个 API 端点：概览卡片、预算明细表、CRUD、进度条
- 新建 `frontend/templates/finance_budget.html`
- 新建 `frontend/static/js/finance_budget.js`
- 实际金额实时聚合 FinanceIncome/FinanceExpense 分组数据
- 进度条：≤100% 绿色 / >100% 红色

### 3. 利润表模块

**背景**: 系统缺少跨模块营收/支出聚合报表，无法一键查看经营利润。

**改动**：
- 新建 `backend/routers/finance_profit.py` — 6 个 API 端点
- 营收聚合跨 5 张表：Sale（售课）、MembershipCard（会籍卡）、Recharge（充值）、ProductSale（零售）、FinanceIncome（其他收入）
- 支出聚合跨 2 张表：FinanceExpense（支出）、ProductSale（商品成本）
- 概览卡：总收入、总支出、净利润、利润率
- 营收构成表 + 支出构成表（含占比）
- 环比（本月 vs 上月）+ 同比（本月 vs 去年同月）
- CSV 导出支持

### 4. 数据分析看板

**背景**: 经营数据分散在各模块，缺少可视化分析工具。

**改动**：
- 新建 `backend/routers/analytics.py` — 8 个 Chart.js JSON 端点
- 8 张图表：营收趋势（Line）、营收构成（Doughnut）、支出趋势（Line）、收支对比（Grouped Bar）、会员增长（Bar）、进场高峰时段（Bar）、Top 10 课程（Horizontal Bar）、支付方式分布（Doughnut）
- 新建 `frontend/templates/analytics.html` — 响应式网格布局
- 新建 `frontend/static/js/analytics.js` — `Promise.all` 并行加载 + `destroyAll()` 防 canvas 泄漏
- 支持近1月/近3月/近12月范围切换

### 6. 功能配置清单（Feature Flag）系统

**背景**: 各健身房业务需求不同，有的功能冗余、有的需要强化。之前要禁用某个功能必须改代码（删路由、删导航），无法快速定制。需要一套配置体系，让客户通过编辑配置文件就能启用/禁用功能模块。

**方案**: YAML 配置 + Python 注册表 + 条件路由 + 动态导航

**改动**:

1. **`backend/features.yaml`（新增）** — 客户可编辑的 27 项功能开关配置文件，按 6 大分组（会员/课程/进场/员工/财务/系统）组织，每项带中文注释：
   ```yaml
   member: true      # 设为 false 则隐藏导航 + 页面 404 + API 不可用
   ```

2. **`backend/feature_registry.py`（新增）** — 功能注册表，职责：
   - `FEATURE_DEFS` 定义全部 27 个功能的元数据（label/group/icon/page_path/routers/resources）
   - `NAV_TREE` 嵌套结构匹配 6 个侧边栏分组（含子分组如业绩总览）
   - `FeatureRegistry` 类：加载 YAML、查询启用状态、过滤导航树、返回活跃路由列表
   - 模块级单例 `registry = FeatureRegistry()`
   - 性能子功能（`performance_sales` 等 4 项）跟随主 `performance` 开关
   - `_ALWAYS_ENABLED` 确保 auth/dashboard/mcp 始终注册
   - 配置文件不存在/格式错误 → 全功能回退（向后兼容）

3. **`backend/app.py`（修改）** — 条件注册路由：
   - 模板全局变量注入 `_nav_tree` / `_group_names`
   - 所有页面路由包裹 `if registry.is_enabled("xxx"):` 条件
   - 所有 API 路由包裹 `if registry.is_enabled("xxx"): app.include_router(...)`
   - `_RESOURCE_CN_MAP` 改为 `registry.get_active_resource_map()`

4. **`frontend/templates/base.html`（修改）** — 导航菜单从硬编码改为 Jinja2 动态渲染：
   ```jinja
   {% for group in _nav_tree() %}
   <button onclick="toggleGroup('{{ group.group }}')">...{{ group.group }}...</button>
   <div id="group-{{ group.group }}">
     {% for item in group["items"] %}
     <a href="{{ item.path }}">{{ item.label }}</a>
     {% endfor %}
   </div>
   {% endfor %}
   ```
   - 注入 `window._GROUP_NAMES` 供 JS 侧边栏状态恢复使用

5. **`frontend/static/js/app.js`（修改）** — `GROUP_NAMES` 从模板注入的 `window._GROUP_NAMES` 读取，不再硬编码

6. **`requirements.txt`（修改）** — 添加 `PyYAML>=6.0`

**设计要点**:
- 客户只需编辑 `features.yaml`，不需要改任何 Python 代码
- 设为 `false` 后：导航隐藏 + 页面路由 404 + API 路由不注册
- 配置文件缺失/错误时自动启用所有功能（零成本回退）
