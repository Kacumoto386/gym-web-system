# 鼠小弟健身管理系统 网页版方案设计文档

## V3.0.1 | 2026-05-08

---

## 1. 项目概述

### 1.1 项目背景

从原桌面版（V2.16.x，Excel + Tkinter）迁移至 Web 版本，解决桌面版多机部署困难、数据无法实时共享、界面不够现代等问题。

### 1.2 项目目标

- 实现全功能 Web 端健身管理系统
- 支持多人同时在线操作
- 一键部署，零配置启动
- 保留全部原有业务逻辑和数据

### 1.3 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| 后端框架 | FastAPI | 高性能异步框架，自动 OpenAPI 文档，简洁 Pythonic |
| ORM | SQLAlchemy 2.x | 成熟稳定，支持多种数据库 |
| 数据库 | SQLite | 单机部署最简方案，无需额外服务 |
| 前端 | Jinja2 + HTMX + Tailwind CSS | 极简前端方案，无需 Node.js/构建工具 |
| 认证 | JWT + Cookie | 无状态认证，适配 HTMX 无 JS 场景 |
| 密码 | bcrypt (5.x) | 直接调用 bcrypt 库，避免 passlib 兼容性问题 |

## 4. API 设计

### 4.1 路由设计原则

1. **双模端点**：每个业务模块同时提供 JSON API 和 HTMX HTML 片段端点
2. **命名规范**：`/api/{resource}`（JSON）/ `/api/{resource}/table`（HTML 片段）
3. **认证**：所有 `/api/*` 和页面路由（除了 `/auth/*`）均需 JWT 认证
4. **操作日志**：POST/PUT/DELETE 请求自动记录到 operation_logs 表

## 5. 前端设计

### 5.1 技术栈

- **HTML 模板**: Jinja2（服务端渲染）
- **动态交互**: HTMX 2.x（属性驱动的 AJAX）
- **样式**: Tailwind CSS（CDN 引入，无需构建）
- **客户端脚本**: 仅登录页有少量 JS，其余零 JS

### 5.2 页面结构

所有页面继承自 `base.html`，包含：
- 左侧边栏导航（固定 224px，深蓝背景，图标+文字竖排）
- 主内容区（`{% block content %}`）
- 导航分组：核心业务、数据分析（可折叠子菜单）、扩展业务、辅助功能
- HTMX 配置（错误处理）

### 5.3 设计特点

- **零 JS 架构**：除登录认证外，所有页面操作通过 HTMX 属性完成
- **无刷新 CRUD**：增删改用 `hx-delete` / `hx-post`，结果刷新表格
- **搜索即时响应**：搜索框绑定 `hx-trigger="keyup changed delay:300ms"`
- **统一错误处理**：HTMX `beforeSwap` 事件拦截 4xx 响应
- **侧边栏导航**：左侧固定竖排布局，图标+文字，当前页高亮，滚动友好

## 6. 认证设计

### 6.1 流程

```
用户 ─→ /auth/login（输入用户名密码）
       ─→ POST /auth/token（验证密码）
          ├─ 失败: 返回 401，显示错误信息
          └─ 成功: 返回 JWT token + Set-Cookie

后续请求 ─→ Cookie 携带 access_token
           ├─ 中间件验证 JWT
           ├─ 无效: 302 重定向到 /auth/login
           └─ 有效: 正常处理请求
```

### 6.2 JWT 配置

- 算法: HS256
- 有效期: 24 小时
- 存储: HttpOnly Cookie（防 XSS）
- 密钥: 应用中硬编码（生产环境应改为环境变量）

### 6.3 密码安全

- 哈希算法: bcrypt（salt 自动生成）
- 库: bcrypt 5.x（纯 Python 实现，避免 C 扩展编译问题）
- 初始化: 首次运行 `/auth/setup` 创建管理员（默认 admin/admin123）

## 7. 操作日志设计

### 7.1 实现方式

- **中间件拦截**：FastAPI middleware 拦截所有 POST/PUT/DELETE
- **自动解析**：从路径 `/api/{resource}/{id}` 提取资源名和 ID
- **写入时机**：仅记录成功操作（2xx 响应）

### 7.2 日志字段

| 字段 | 说明 |
|------|------|
| operator | 操作人（从 JWT token 提取） |
| action | 操作类型（create/update/delete） |
| resource | 资源名称（如"会员""课程"） |
| resource_id | 资源 ID |
| detail | 完整请求路径和方式 |
| ip_address | 客户端 IP |
| created_at | 操作时间 |

## 8. 数据迁移方案

### 8.1 迁移策略

1. 读取 Excel 原始文件（健身房管理系统.xlsx）
2. 按 Sheet 逐表映射到 SQLAlchemy ORM 模型
3. 自动处理数据类型转换（Excel 日期 → Python date）
4. 自动生成缺失的唯一编号（如空 member_id → M+日期+序号）
5. 数据验证（重复编号、非法格式等）

### 8.2 迁移工具

`backend/migrations/migrate_from_excel.py`

```bash
python backend/migrations/migrate_from_excel.py --excel /path/to/健身房管理系统.xlsx
```

### 8.3 迁移统计（现状）

- 总记录数: 401 条
- 有数据的表: 21 张
- 迁移成功率: 100%

## 9. 部署方案

### 9.1 一键启动

`start.bat` 执行流程：
1. 检查 Python 3.12 环境
2. pip 安装依赖（fastapi, uvicorn, sqlalchemy, jinja2 等）
3. 检查/创建默认管理员（admin/admin123）
4. 启动 uvicorn 服务器
5. 浏览器访问 `http://127.0.0.1:8000`

### 9.2 生产部署建议

| 组件 | 生产环境建议 |
|------|-------------|
| Web 服务器 | uvicorn + nginx 反向代理 |
| 数据库 | 迁移到 PostgreSQL / MySQL |
| 环境变量 | SECRET_KEY 从环境变量读取 |
| HTTPS | Let's Encrypt SSL 证书 |
| 进程管理 | supervisor / systemd 守护 |
| 日志 | 改用 loguru 或标准 logging |

## 11. 实用工具

|------|------|
