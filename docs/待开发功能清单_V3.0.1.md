## 3️⃣ 商品零售购物车模式（对应单机版 V2.15.5）

**来源文档**: `docs/商品零售购物车模式方案_V2.15.5.md`

### 现状
当前 `/products` 是简单的商品管理 + 单品零售登记，无购物车概念。

### 需要改造的功能
1. **多商品购物车**：选择多种商品加入购物车，统一结算
2. **优惠金额**：结算前可设置优惠金额（比例分摊到各商品）
3. **储值支付**：会员可选择用储值余额支付

### UI 布局建议（三栏）
```
├─ ① 商品选择区 ──── ② 购物车列表 ──── ③ 结算区 ──┤
│ 商品下拉框         表格：商品名/数量    会员选择        │
│ 数量输入           单价/小计          合计金额         │
│ [加入购物车]       [删除] [清空]      优惠金额         │
│                                        应付金额         │
│                                        支付方式         │
│                                        ☑ 使用储值       │
│                                        [确认结算]       │
```

### 改造文件
| 文件 | 说明 |
|------|------|
| `frontend/templates/products.html` | 重构为三栏购物车布局 + 购物车 JavaScript 逻辑 |
| `backend/routers/product.py` | 新增批量结算 API：`POST /api/product-sales/batch` |
| `backend/routers/member.py` | 新增储值余额查询 API：`GET /api/members/{member_id}/balance` |

### 数据流
```
加入购物车 (前端 JS 维护 cart[]) → 确认结算
→ POST /api/product-sales/batch (批量写入)
→ 扣减库存 (product.stock -= quantity)
→ 扣减储值 (如使用储值支付)
→ 返回结算成功
```

## 4️⃣ 梯度提成模块（对应单机版 V2.15.5）

**来源文档**: `docs/梯度提成模块改造方案_V2.15.5.md`

### 现状
ORM 表 `commission_tier` 已存在且有 7 条默认数据，但无管理页面和提成计算功能。

### 需要实现的功能
1. **梯度规则管理页面**：CRUD（新增/编辑/删除梯度规则）
2. **金额预览/试算**：输入金额查看分段累进计算结果
3. **员工佣金计算**：在员工详情中展示基于梯度的售课/上课提成
4. **分段累进引擎**：将销售额/上课节数拆分到各档区间分别计算

### 涉及新增文件
| 文件 | 说明 |
|------|------|
| `frontend/templates/commission_tiers.html` | 梯度配置管理页面 |
| `backend/routers/commission.py` | 梯度规则 CRUD API + HTML 片段 |

### 默认规则（已有数据）
**售课提成**：0~3万(5%) → 3~8万(8%) → 8~15万(10%) → 15万以上(12%)
**上课提成**：0~30节(5元/节) → 30~60节(8元/节) → 60节以上(10元/节)

### 两种计算模式
```python
# 全额匹配：全部销售额按所在档位提成率计算
commission = amount * rate  # 简单但不够公平

# 分段累进（推荐）：各区间分别计算再累加
segments = [
    {"range": "0~3万", "amount": 30000, "rate": "5%", "commission": 1500},
    {"range": "3~8万", "amount": 50000, "rate": "8%", "commission": 4000},
    ...
]
total = sum(s["commission"] for s in segments)  # 更公平
```

## 实施顺序建议

**Phase 1（高优先级）**：会籍卡联动进场核销（#5/#9）+ 业绩总览看板

- 锦上添花，不影响核心业务流程
